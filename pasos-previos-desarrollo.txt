Secuencia previa completa antes del prompt del primer hito
1.	Decide desde qué superficie vas a trabajar con Codex: app de Windows, extensión de IDE, CLI, o combinación de varias. En Windows, la documentación oficial indica que la opción más sencilla es la app, y además la CLI y la extensión comparten la misma capa de configuración. (OpenAI para desarrolladores)
2.	Deja operativo el entorno Rust local hasta el punto de poder trabajar con Cargo sin fricción. Cargo es el gestor oficial del ecosistema Rust y será la base de compilación, test, documentación y empaquetado. (doc.rust-lang.org)
3.	Crea el repositorio Git desde el minuto cero y decide si será repo nuevo o si vivirá dentro de un monorepo mayor. Codex recomienda usar checkpoints de Git antes y después de cada tarea para poder revertir cambios con facilidad. (OpenAI para desarrolladores)
4.	Fija la raíz real del proyecto: carpeta exacta, nombre definitivo del repo y convención de rutas. Esto es importante porque Codex carga configuración y contexto por proyecto y por directorio. (OpenAI para desarrolladores)
5.	Decide que el proyecto arranca como workspace de Cargo, aunque al principio solo tenga un crate funcional. Los workspaces están pensados para gestionar varios paquetes relacionados bajo una misma raíz y te ahorran una migración estructural más adelante. (doc.rust-lang.org)
6.	Define la estructura de alto nivel del repo sin rellenarla todavía: raíz, crates/, docs/, tests/, benches/, scripts/, .codex/, workflows, etc. No es para escribir contenido aún; es para fijar el esqueleto y que Codex no improvise jerarquías. Esto encaja especialmente bien con workspaces de Cargo. (doc.rust-lang.org)
7.	Decide los crates previstos desde el inicio, aunque algunos empiecen vacíos: por ejemplo núcleo, TUI, API, infraestructura y utilidades internas. El objetivo aquí no es implementarlos, sino dejar claro el mapa de crecimiento del workspace. (doc.rust-lang.org)
8.	Crea el AGENTS.md raíz del proyecto. Codex lee AGENTS.md antes de empezar a trabajar y combina instrucciones globales y específicas del proyecto, así que este archivo será tu contrato permanente con el agente. (OpenAI para desarrolladores)
9.	Decide la política de instrucciones jerárquicas: qué irá en el AGENTS.md raíz y qué cosas, si hiciera falta, podrían ir más adelante en subdirectorios. La guía oficial explica que Codex carga esos archivos en capas desde la raíz hacia el directorio actual. (OpenAI para desarrolladores)
10.	Crea .codex/config.toml del proyecto para no depender solo de tu configuración global. Codex permite configuración por usuario y también overrides por proyecto; cuando encuentra varias, la más cercana al directorio actual tiene prioridad. (OpenAI para desarrolladores)
11.	Decide en ese config del proyecto el modelo, el nivel de aprobación y el modo de trabajo por defecto. La documentación de Codex distingue claramente configuración por proyecto, control de modelo y política de ejecución. (OpenAI para desarrolladores)
12.	Deja definida tu política de seguridad para Codex antes de tocar código: qué puede ejecutar solo, qué debe pedir permiso y qué no debe tocar. Las reglas están precisamente para controlar qué comandos puede ejecutar fuera del sandbox, y la documentación las marca además como experimentales, así que conviene tratarlas de forma explícita. (OpenAI para desarrolladores)
13.	Crea la carpeta .codex/rules/ y decide sus categorías, aunque al principio solo pongas reglas mínimas. No escribas prompts del hito 1 sin haber decidido antes qué comandos son seguros, cuáles requieren confirmación y cuáles quedan vetados. (OpenAI para desarrolladores)
14.	Decide si vas a usar MCP o integraciones externas desde el principio o no. No hace falta activarlo ya, pero sí decidir si el proyecto va a depender de herramientas externas conectadas a Codex, porque esa configuración también vive en config.toml. (OpenAI para desarrolladores)
15.	Prepara el README.md base antes del primer hito, aunque todavía esté incompleto. No para describir la implementación, sino para fijar propósito, alcance, mapa del repo, forma de validar y estado del proyecto. Además, Rust/Cargo tienen una relación muy directa entre documentación y validación del código. (doc.rust-lang.org)
16.	Crea la carpeta docs/ y separa desde ya la documentación por dominios: visión, arquitectura, decisiones, testing, operaciones y hitos. Esto no viene impuesto por Cargo, pero te va a evitar mezclar especificación, decisiones y manual operativo dentro del README. El propio ecosistema de Rust favorece una documentación muy integrada con el flujo de desarrollo. (doc.rust-lang.org)
17.	Escribe primero la visión del producto a nivel de sistema, no a nivel de UI. Antes del hito 1 debes dejar claro qué clase de aplicación es, qué papel tendrá la API integrada y qué no entra en alcance todavía. OpenAI recomienda que el trabajo con Codex se formule con objetivo, contexto, restricciones y condición de terminado; esta visión te servirá de base para eso. (OpenAI para desarrolladores)
18.	Escribe después una arquitectura objetivo v0: separación entre core, TUI, API y bordes, incluso si luego cambias detalles. Para un proyecto Rust con crecimiento por hitos, te interesa fijar pronto la topología del workspace. (doc.rust-lang.org)
19.	Abre una carpeta de decisiones técnicas y acuerda que cualquier decisión importante se registra ahí desde el día 1. Esto evita que Codex y tú reabráis debates cerrados en hitos posteriores. La necesidad viene de cómo Codex reutiliza contexto de proyecto y de cómo conviene darle instrucciones estables y acumulativas. (OpenAI para desarrolladores)
20.	Define el roadmap completo de hitos antes de escribir el primero. No hace falta detallar aún cada tarea, pero sí el mapa: orden, dependencias, qué desbloquea cada hito y qué queda fuera. Eso encaja con la recomendación de trabajar con metas y criterios de finalización claros. (OpenAI para desarrolladores)
21.	Crea una plantilla fija para los documentos de hito y comprométete a no salirte de ella. Cada hito debería tener los mismos bloques de alcance, pruebas, regresión, documentación y definición de hecho. Así Codex trabajará siempre sobre un contrato homogéneo. (OpenAI para desarrolladores)
22.	Define la estrategia oficial de testing antes del hito 1. Cargo soporta pruebas unitarias, de integración y de documentación; además, la organización estándar separa src y tests/, y cada archivo en tests/ se trata como crate de integración independiente. (doc.rust-lang.org)
23.	Decide qué categorías de pruebas son obligatorias por hito: unitarias, integración, doc tests, E2E, rendimiento, carga, concurrencia, estrés y regresión. No tienes que implementarlas aún; tienes que dejar decidido cuáles entran siempre y cuáles solo en hitos concretos. Cargo ya te cubre una base oficial de test y documentación; lo demás lo construirás encima. (doc.rust-lang.org)
24.	Define la política de linting y formato antes del primer prompt. Clippy es la herramienta oficial de lints para detectar errores comunes y mejorar el código Rust, y cargo clippy es el flujo recomendado; del mismo modo, la normalización de formato debe quedar fijada desde el inicio. (doc.rust-lang.org)
25.	Decide cómo vas a generar y revisar documentación del código. cargo doc genera la documentación local del paquete y dependencias, así que conviene dejar ya pactado si la documentación forma parte del “done” de cada hito. (doc.rust-lang.org)
26.	Define la política de features de Cargo antes de que aparezcan las primeras opciones técnicas. Cargo soporta features para compilación condicional y dependencias opcionales; si no pactas pronto cómo se van a usar, luego se vuelven caóticas. (doc.rust-lang.org)
27.	Decide los comandos oficiales del proyecto: compilar, ejecutar, testear, documentar y validar. Cargo ya marca la base de estos flujos con comandos estándar como cargo run, cargo test y cargo doc; tú debes decidir cuáles se convierten en la “verdad oficial” del repo. (doc.rust-lang.org)
28.	Define la política de checkpoints y branches por hito antes de pedir el primer cambio a Codex. La recomendación oficial de usar checkpoints de Git es especialmente importante cuando el agente puede editar archivos y ejecutar tareas sobre el repo. (OpenAI para desarrolladores)
29.	Prepara la base de CI/CD aunque sea mínima: qué validaciones serán obligatorias desde el primer hito y cuáles llegarán después. El ecosistema Cargo ya cubre de forma natural build, test y documentación, así que tu trabajo aquí es fijar qué bloquea avance y qué no. (doc.rust-lang.org)
30.	Decide si crearás skills de Codex ahora o más tarde. Son opcionales; lo importante antes del hito 1 es decidir si vas a mantener todo mediante AGENTS.md y prompts estructurados o si ya hay flujos repetitivos que merecen skill propia. (OpenAI para desarrolladores)
31.	Solo cuando todo lo anterior exista, redacta la plantilla maestra de prompt para hitos. La guía de prompting para Codex insiste en que el modelo responde mejor cuando el contexto persistente vive en AGENTS.md y el prompt del trabajo se centra en objetivo, alcance, restricciones y condición de terminado. (OpenAI para desarrolladores)
32.	Y solo después de eso redacta el prompt del hito 1. En ese momento ya tendrás repo, reglas, arquitectura, testing, documentación y contrato de trabajo listos, así que el primer hito podrá centrarse de verdad en construir y no en improvisar fundamentos. (OpenAI para desarrolladores)
